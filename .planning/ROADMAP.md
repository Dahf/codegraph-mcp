# Roadmap: CodeGraph MCP

## Overview

Six phases that build a semantic code knowledge base from the ground up. The pipeline flows naturally: stand up the server and storage layer, then parse code into the graph, then embed it into vectors, then wire up MCP query tools, then add re-indexing automation, then cap it with a web dashboard. Each phase produces something the next phase consumes — nothing is built speculatively.

## Phase Details

### Phase 1: Foundation
**Goal**: A running MCP server with Streamable HTTP transport, initialized storage backends, and a working repository management API
**Depends on**: Nothing (first phase)
**Requirements**: MCPS-01, OPS-01, OPS-03
**Success Criteria** (what must be TRUE):
  1. MCP server starts and responds to Streamable HTTP connections (no SSE)
  2. FalkorDB and LanceDB connections initialize without error on server start
  3. User can add, remove, and list repositories via API or config file
  4. Health endpoint returns uptime, indexed repo count, and total symbol count
**Plans**: 3 plans

Plans:
- [x] 01-01-PLAN.md — TypeScript project scaffolding, MCP SDK setup, Streamable HTTP transport
- [ ] 01-02-PLAN.md — FalkorDB, LanceDB, and Ollama adapter setup with fail-fast startup validation
- [ ] 01-03-PLAN.md — Repository management API (add/remove/list) and health endpoints

### Phase 2: Parsing & Graph Storage
**Goal**: The indexing pipeline parses all 5 languages, extracts structured symbols and call-graph relationships, and stores them in FalkorDB
**Depends on**: Phase 1
**Requirements**: PARS-01, PARS-02, PARS-03, PARS-04, PARS-05, PARS-06, PARS-09, STOR-01, STOR-03
**Success Criteria** (what must be TRUE):
  1. Running index on a TypeScript, Python, Rust, Go, or C++ file produces function/class/type/import nodes in FalkorDB
  2. Call-graph edges exist in the graph: functions that call other functions are connected
  3. Cross-file relationships (imports, calls across files) are captured in the graph
  4. Graph schema includes Files, Functions, Classes, Types, Imports, and Calls edge types
  5. Parse errors on a single file do not abort indexing of the remaining files
**Plans**: 4 plans

Plans:
- [x] 02-01-PLAN.md — TreeSitter packages, type definitions, language registry, cloner, walker, IndexPipeline skeleton
- [ ] 02-02-PLAN.md — TypeScript/JavaScript and Python extractors (functions, classes, types, imports)
- [ ] 02-03-PLAN.md — Rust, Go, C++ extractors with graceful degradation on parse errors
- [ ] 02-04-PLAN.md — Call-graph extraction, FalkorDB graph writer, index REST endpoints

### Phase 3: Embedding & Vector Storage
**Goal**: Parsed code is chunked at semantic boundaries and embedded locally via Ollama, with all embeddings stored in LanceDB
**Depends on**: Phase 2
**Requirements**: PARS-07, PARS-08, STOR-02
**Success Criteria** (what must be TRUE):
  1. Running index on a repository stores one embedding per function or class in LanceDB (not per file)
  2. Embeddings are generated by the CodeRankEmbed model running locally via Ollama
  3. LanceDB holds a queryable vector index after indexing completes
**Plans**: 2 plans

Plans:
- [ ] 03-01-PLAN.md — Adapter extensions (OllamaAdapter.embed, LanceDBAdapter write/delete), semantic chunker, IndexResult types
- [ ] 03-02-PLAN.md — Embedder orchestrator with p-limit concurrency, Pipeline Stage 5 integration, app layer wiring

### Phase 4: MCP Query Tools
**Goal**: All three MCP tools are live and return accurate results by combining graph traversal and vector similarity search
**Depends on**: Phase 3
**Requirements**: MCPS-02, MCPS-03, MCPS-04
**Success Criteria** (what must be TRUE):
  1. Claude Code or another MCP client can call Semantic Search with a natural language query and receive relevant code snippets from indexed repos
  2. Symbol Lookup called with a function or class name returns the source code and file location of that symbol
  3. Context Bundle called with a task description returns a pre-assembled set of relevant code context ready for an AI to consume
  4. All three tools function across multiple indexed repositories in a single query
**Plans**: TBD

Plans:
- [ ] 04-01: Query engine — vector similarity search against LanceDB with result ranking
- [ ] 04-02: Graph query layer — symbol lookup and call-graph traversal via FalkorDB
- [ ] 04-03: Semantic Search and Symbol Lookup MCP tool implementations
- [ ] 04-04: Context Bundle assembly strategy and MCP tool implementation

### Phase 5: Re-indexing & Operations
**Goal**: The knowledge base stays current automatically — git commits trigger incremental re-indexing that only processes changed files
**Depends on**: Phase 4
**Requirements**: PARS-10, OPS-02
**Success Criteria** (what must be TRUE):
  1. Making a commit to a registered repository triggers re-indexing automatically (via git hook)
  2. Re-indexing only processes files that changed since the last indexed commit, not the entire repository
  3. Re-indexing completes and the updated symbols are queryable without restarting the server
**Plans**: TBD

Plans:
- [ ] 05-01: Incremental indexing engine — file change detection, delta extraction, upsert into graph and vector stores
- [ ] 05-02: Git hook installer and post-commit webhook handler

### Phase 6: Web Admin Dashboard
**Goal**: A web dashboard lets users see repository status, monitor indexing, trigger manual re-indexes, and view server health — all without touching the CLI
**Depends on**: Phase 5
**Requirements**: UI-01, UI-02, UI-03, UI-04
**Success Criteria** (what must be TRUE):
  1. Opening the dashboard URL shows a list of all registered repositories with their current index status
  2. Each repository row shows the last indexed commit and indexing progress during active indexing
  3. User can click a button on any repository row to trigger a manual re-index
  4. Dashboard shows a server health panel with total symbols, total embeddings, and repo count

- [ ] 06-02: Dashboard frontend — repository list, status display, and re-index button
- [ ] 06-03: Health stats panel and live indexing progress display
